Why

SSIA

データエンジニアは「並列」を毎日触っているのに、実装の中身は意外と曖昧にしたまま動かしていることが多い。

  • Airflow の parallelism / max_active_runs をどう決めるか
  • BigQuery のジョブは「投げて待つ」非同期。.result() の前後で何が起きているか
  • dbt の threads を上げたら速くなるけど、上げすぎると詰まる
  • Python スクリプトで requests を並列化したい

これらを 「スレッド」「プロセス」「非同期」のどれで解いているかを区別できると、設計の解像度が一段上がる。

What

1. 並列の 3 形態（超ざっくり）

形態 同時性 向いている処理
マルチプロセス 真の並列（CPU コア数まで） CPU バウンド（重い計算、前処理）
マルチスレッド OS が時分割（Python は GIL あり） I/O バウンド（API、DB、ファイル）
非同期 (async/await) シングルスレッド + 協調的切替 大量の I/O（HTTP, DB）

Python のデータエンジニアが触るのは、ほぼ I/O バウンド。だから「スレッド」or「非同期」が中心。

2. 同期 vs 非同期（1枚絵）

同期（順番に処理） 

[ジョブA投げる]→[A待つ]→[B投げる]→[B待つ]→[C投げる]→[C待つ]

非同期（投げて、待つのは後でまとめて） 

[A投げる][B投げる][C投げる]→[A待つ B待つ C待つ]

データエンジニアが触るほぼ全ての API（BigQuery、GCS、HTTP）は 「投げる」と「待つ」を分けられる非同期 I/F を持っている。

3. BigQuery を例に 

from google.cloud import bigquery

client = bigquery.Client()

# 1. ジョブを投げる（非同期 / すぐ返る）
job = client.query("SELECT COUNT(*) FROM `proj.ds.tbl`")
print(job.job_id, job.state)   # 'RUNNING'

# 2. ここで他の仕事を進められる

# 3. 結果が要るタイミングで待つ
rows = job.result()             # ここでブロック

「client.query() ＝ジョブ投入」「job.result() ＝待ち」。これを意識すると、複数クエリを同時投入 → まとめて待つ設計に切り替えられる。

4. Airflow / Composer の並列パラメータ

Airflow には「並列度の蛇口」が階層的にある。どれを上げ下げしているかを意識する。

パラメータ スコープ
parallelism クラスタ全体で同時実行できるタスク数
dag_concurrency 1 DAG 内で同時実行できるタスク数
max_active_runs 同じ DAG で同時に走れる run の数
pool 共有リソース（外部 DB 等）を奪い合う粒度の制御
task_concurrency 1 タスク種が同時に走れる数

「タスクが詰まっている」ときは、上から順に「どこで詰まっているか」を当てる。

5. dbt の threads

  • dbt は 依存グラフを並列に解いていく
  • threads は DWH に 同時投入する接続数
  • 上げると速いが、DWH 側のスロット / コネクション枠を奪うので、他のジョブを詰まらせることがある

「dbt が早くなった = 他のクエリが遅くなる」ことがある。BigQuery なら slot 競合、Snowflake なら warehouse 同時実行に注意。

6. 並列で起きがちな事故

事故 原因 対処
同じテーブルに同時 INSERT 並列度を上げすぎ パーティション分割 / シリアライズ
API レート制限に引っかかる 同時リクエスト過多 バックオフ + 並列度の上限制御
デッドロック 複数リソースを違う順序で取得 取得順を統一
メモリ枯渇 並列ごとにバッファが積み上がる バッチサイズを下げる
ロギングが混線 複数スレッドが同じ stdout スレッド ID をログに含める

7. 効きどころ

  • Composer の DAG 設計で、「直列で書くべきか / 並列で書くべきか」を根拠を持って判断できる
  • BQ ジョブを Python から大量投入するときに、job.result() の位置を最適化して全体時間を短縮できる
  • dbt の threads 調整で、速度と他ジョブへの影響を天秤にかけられる
