Engineering Networking TCP/IP BigQuery Composer

Why

SSIA

データエンジニアは BigQuery、GCS、S3、Looker API、Composer のタスクなど、ほぼ全部「ネットワーク越し」に動かしている。

  • 「Connection reset by peer で DAG が落ちた」
  • 「bq load が途中で 503 を返した」
  • 「Looker のクエリだけ妙にタイムアウトする」

これらが「ネットワークのどの層で何が起きたのか」を読めるかどうかで、復旧スピードが何倍も変わる。

What

1. TCP/IP の超ざっくり地図

レイヤ 役割 データエンジニアが触る場面
アプリ層（HTTP/gRPC） 何を送るか bq CLI、gcloud、REST API、Looker API
トランスポート層（TCP） 確実に届ける コネクション確立、再送、輻輳制御
インターネット層（IP） どこへ届けるか ルーティング、サブネット、VPC
リンク層 物理 / L2 ほぼ触らない

ポイント：エラーメッセージが「どの層」のものかを当てると原因の範囲が一気に絞れる。

2. TCP の挙動で知っておくべき4つ

(1) 3-way handshake（接続を張る）

  • SYN → SYN-ACK → ACK の 3 回で接続を確立する
  • ここで失敗すると Connection refused（相手プロセスが居ない）/ timeout（そもそも到達しない）

(2) 再送（ロスっても粘る）

  • パケットが落ちると TCP は勝手に再送してくれる
  • だから「ネットワーク不調＝必ず即エラー」ではなく、遅いまま動き続けることがある

(3) コネクション切断

  • 正常終了は FIN、異常終了は RST
  • Connection reset by peer ＝相手側が RST を投げてきた、と読める

(4) タイムアウト

  • TCP のタイムアウトとアプリ側（HTTP クライアント、bq CLI 等）のタイムアウトは別物
  • アプリ側のタイムアウトのほうが短いことが多く、先に切るのはアプリ側

3. データエンジニアあるある

症状 TCP/IP 視点での読み
Connection refused 相手プロセス未起動 or ポート間違い。サービス疎通から確認
i/o timeout / connect timeout 到達はできていない疑い。Firewall / VPC-SC / NAT を確認
Connection reset by peer 相手が途中で切った。LB の idle timeout、サーバ側 OOM 等
503 Service Unavailable 到達はできた。アプリ層の問題（負荷、デプロイ中、レート制限）
bq load が途中で失敗 リトライで救えるケース多い。冪等性を担保した上で retry policy を設定

4. デバッグの最小ツールキット 

# 名前解決＋疎通確認
ping bigquery.googleapis.com           # ICMP（FW で塞がれていると無反応）
nc -vz bigquery.googleapis.com 443     # TCP の到達確認だけ
curl -v https://bigquery.googleapis.com  # アプリ層まで貫通

# どこで詰まっているかの切り分け
traceroute bigquery.googleapis.com     # 経路の可視化

切り分けは「内 → 外」「下層 → 上層」の順で。nc で TCP 通れば下層は OK、と一段ずつ確定する。

5. データ基盤でとくに効く場面

  • Composer (Airflow) の HttpOperator が時々落ちる → リトライポリシーと exponential backoff を設定する妥当性を判断できる
  • BigQuery への bq load で Connection reset → サーバ側の問題かネットワークかを切り分けて、retry で吸えるか判断できる
  • VPC-SC で外部 API が叩けない → IP 層の到達性問題と理解できれば、egress 設計の議論に乗れる
