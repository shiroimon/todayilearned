枝を覗くか、引き寄せるか（fetch vs pull）

リモートの状態を取ってくる際の2つのコマンド。git pull ≒ git fetch + git merge（または git rebase）。

コマンド やること 作業ブランチへの影響
git fetch リモートの最新情報を取ってくるだけ なし
git pull fetch した上で現在のブランチへマージ／リベース あり（ファイルが更新される） 
remote -[main] branch -----C---D---------------------->
                                      ↑/fetch  ↑/pull
local  -[main] branch ----B-----------|---------|---->
                                      |         └→ 取り込んで merge/rebase
                                      └→ origin/main の参照だけ更新
  • 「他人のブランチを覗くだけ」「リモートの状態を知りたいだけ」 → fetch
  • 「自分のブランチを最新に追いつかせたい」 → pull

git pull origin でも他人のリモートブランチは取れている

git pull origin を実行すると、内部では git fetch origin（= デフォルト refspec +refs/heads/*:refs/remotes/origin/* により全リモートブランチを取得）が走り、そのあとに現在のブランチへマージされる。

つまり「pull したのに git branch -r に新ブランチが出てこない」場合は、pull した後にリモートで新ブランチが push されたタイミングの問題である可能性が高い。

# もう一度 fetch すれば最新化される
$ git fetch origin --prune
$ git branch -r | grep <ブランチ名>

pull で rebase したい場合

--rebase を付けると、② が merge ではなく rebase になる。マージコミットを残さず履歴を一直線に保ちたい人向け。

# 単発
$ git pull --rebase origin main

# このリポジトリだけデフォルト化
$ git config pull.rebase true

# 全リポジトリでデフォルト化
$ git config --global pull.rebase true
式 意味
git pull git fetch + git merge
git pull --rebase git fetch + git rebase

--prune で残骸を剪定（タイトル回収）

--prune をつけると、リモートで削除されたブランチをローカルの追跡からも消してくれる。git branch -r に居残る残骸の剪定。

# 一度だけ
$ git fetch --prune

# 常に --prune するようグローバル設定（推奨）
$ git config --global fetch.prune true

枝から枝を（名前変更）

Tip

まだ push していないなら git branch -m <new> 1 発で終わる。リモートに何も出ていないので後片付けも要らない。 下記の 4 ステップは 既に push 済みのブランチ名を変えたい時の手順。

状況 手数 やること
push 前 1 ステップ git branch -m <new> だけ
push 済み 4 ステップ リネーム → 新名 push → 旧名 削除 → fetch --prune 
# push 前（ローカルだけで完結）
$ git branch --show-current
feature_B
$ git branch -m feature_C        # 名前変更
$ git branch --show-current
feature_C
# → そのまま開発を続行して、初回 push 時に新名で push -u すれば OK

# branch名を[feature_B] → [feature_C] に変えたい（push 済みの場合）

local -[main] branch ------------------------------------------------->
            └-> Aさん[feature_A] branch ------------------------------>
                └-> Bさん[feature_B] branch -> [feature_C] branch ---->

  1. # 現在ブランチ確認
$ git branch
* feature_B
  main

  2. # 現在ブランチ名をリネーム
$ git branch -m feature_C

  3. # 新しいブランチ名をリモートにプッシュ反映
$ git push origin feature_C

  4. # 古いブランチ名はリモートから削除（後片付け）
$ git push origin --delete feature_B
# 自分のローカルの追跡ブランチからも消す（他の開発者は各自 git fetch --prune が必要）
$ git fetch origin --prune

枝先を束ねる（squash & force-with-lease）

開発中は意味論単位で commit / push を繰り返し、PR 提出前に rebase -i でコミットを束ねる運用。レビュアーが「最終的に何が変わるか」だけ見れば済むようになる。

基本運用：意味論単位で commit → push を反復

local  [feature/hoge-0000] ---E1---E2---E3---E4-------->
                               ↓    ↓    ↓    ↓
                              push push push push
remote [feature/hoge-0000] ---E1---E2---E3---E4-------->

PR整理：rebase -i origin/main でスカッシュ → force-with-lease で上書き

main:    A---B---C---D
              \
feature:       E1---E2---E3---E4   ← 整理前（4コミット）

                  ↓ git rebase -i origin/main（squash で1本に）

main:    A---B---C---D
                      \
feature:               E'           ← 1コミットに圧縮（新ID）

                  ↓ git push --force-with-lease origin feature/hoge-0000

remote [feature/hoge-0000] ---E'-------->   ← リモートを上書き
  1. 開発中（意味論単位で何度でも） 
    $ git add <file>
$ git commit -m "WIP: パーサーの分岐追加"
$ git push origin feature/hoge-0000
  2. PR 提出前にコミットを束ねる（origin/main 起点で safe） 
    $ git fetch origin
$ git rebase -i origin/main
#   → エディタで squash / fixup を選んでコミットを統合
  3. リモートを上書き（他人の push を踏まないよう --force-with-lease） 
    $ git push --force-with-lease origin feature/hoge-0000

この運用のチェックポイント

  • rebase -i main よりも rebase -i origin/main が安全

    • ローカル main が古いと、古い基点で rebase してしまい、後で再 rebase が必要になる

  • --force-with-lease は --force より安全

    • リモートが自分の知っている状態と一致する時だけ上書きする
    • 他人が同じブランチに push していたら拒否される（事故防止）

  • レビュー開始後の force push には注意

    • レビュアーが「前回見たコミットからの差分」を追えなくなる
    • その場合は普通に commit / push を続け、最終マージで GitHub の Squash merge を使う選択肢もある

  • スカッシュ後のコミットメッセージは PR タイトル相当に整える（後でログを遡る人のため）

[Merge] 簡単かつ安全（でも履歴汚ねぇ…）

  1. 最新のmain（またはdevelop）ブランチを取得 
    $ git checkout main
$ git pull origin main
  2. 作業ブランチに移動し、マージ 
    $ git checkout feature_A
$ git merge main
    • (if) マージコンフリクトの解消 
      $ git add <conflicted-files>
$ git commit -S -m "memo"
  3. 変更をリモートにプッシュ 
    $ git push origin feature_A

[Rebase] 綺麗な履歴

mainブランチの最新のコミットの後に来るように並べ替えることができる。 これにより、コミット履歴が綺麗な状態で最新コードを取り込むことができる。 一方で、特に共有ブランチに対して行う場合は強制プッシュが必要になる。（他の開発者に影響が出る可能性あり）

  1. $ git checkout main
$ git pull origin main

  2. $ git checkout feature_A
$ git rebase main
    • (if) リベース中にコンフリクトが発生した場合 
      $ git add <conflicted-files>
$ git rebase --continue

      # 中断したい場合
$ git rebase --abort
  3. リモートに強制プッシュ 
    $ git push --force-with-lease origin feature_A

GPG (署名付きコミット) 

                                      　↑/commit :GPG 「わいやで」
-[main] branch　------------------------|---------------->
      └> Aさん[feature_A] branch -------┘
  • GitのGPGは、Gitにおいてコミットやタグなどの操作に署名を付けるための仕組み。
  • GPG（GNU Privacy Guard）は、デジタル署名や暗号化を行うためのツール。
  • GitのGPGを使用することで、コミットやタグが本当に信頼できるものであることを確認することができる。

推奨セット（macOS） 0) 事前：インストール（Homebrew 派）

brew install gnupg pinentry-mac
  1. Git 署名の基本設定
# 署名鍵（できれば40桁のフルフィンガープリント）
git config --global user.signingkey YOUR_GPG_KEY_FPR

# 常にコミット/タグを署名
git config --global commit.gpgsign true
git config --global tag.gpgSign true

# gpg の場所を明示（Homebrew の標準パス）
git config --global gpg.program /opt/homebrew/bin/gpg
  1. gpg-agent のキャッシュを「1日」に
mkdir -p ~/.gnupg
cat > ~/.gnupg/gpg-agent.conf <<'EOF'
pinentry-program /opt/homebrew/bin/pinentry-mac
default-cache-ttl 86400
max-cache-ttl 86400
EOF

# 反映
gpgconf --kill gpg-agent
gpgconf --launch gpg-agent

# 確認
git show --show-signature

# Homebrewで下準備
$ brew install gnupg pinentry-mac
# GPGキーペアの生成
$ gpg --full-generate-key
# 公開鍵の取得
$ gpg --list-keys # $ gpg --list-secret-keys --keyid-format LONG
# Gitに公開鍵を登録:
$ gpg --armor --export YOUR_GPG_KEY_ID | pbcopy
  • GitHubの場合
    • Settings > SSH and GPG keys > New GPG key
    • ヘッダー（-----BEGIN PGP PUBLIC KEY BLOCK-----と-----END PGP PUBLIC KEY BLOCK-----）を含めて貼り付け
# 1) 署名に使う鍵を設定（なるべくフルフィンガープリント推奨）
git config --global user.signingkey YOUR_GPG_KEY_FPR

# 2) コミット（とタグ）を常に署名
git config --global commit.gpgsign true
git config --global tag.gpgSign true

# 3) 必要なら gpg コマンドを指定（環境により gpg / gpg2）
git config --global gpg.program gpg

# 4) gpg-agent のキャッシュを「1日（= 86400秒）」に設定
mkdir -p ~/.gnupg
printf "default-cache-ttl 86400\nmax-cache-ttl 86400\n" >> ~/.gnupg/gpg-agent.conf

# 5) gpg-agent を再起動して設定反映
gpgconf --kill gpg-agent
gpgconf --launch gpg-agent

# 6) 署名が付いているか確認（直近のコミットを見る）
# 出力の中に default-cache-ttl が 86400 などに設定されていれば OK
git show --show-signature

# 署名付きテスト
echo "hello" | gpg --clearsign

$ git archive {add/your_branch_name} `git diff --name-only origin/staging add/shops_referral_sheet --diff-filter=ACMR` -o archive.zip

  • origin/staging と {add/your_branch_name} ブランチの差分を表示
  • --diff-filter=ACMR は、追加(A)、コピー(C)、修正(M)、リネーム(R) されたファイルのみを対象とする

ワークツリー

同じリポジトリで複数の作業ディレクトリを同時に持つ（cd 移動で事足りる）ことができる機能
並列改修でガチャガチャしている時とかに有用。

# 従来：ブランチ切り替えで環境再構築が必要
git switch feature_B/hotfix
# → node_modules再インストール、ビルド再実行、IDEのインデックス再構築...事故ﾘｽｸ高い

# Worktree：即座に別作業に移れる
cd ../hotfix
# → すでに環境が準備済み！
  • 従来：
  • # 機能Aの開発中に緊急バグ修正依頼が来た時の様子

local -[main] branch ---------------------------*---------------->
            └--> ①[feature_A/] branch ---------|--> commit ③------>
            |                                 　|
            └--> ②[feature_B/hotfix] branch ---┘

    # 機能Aの開発中...
git add .
git commit -m "WIP: 機能Aの途中"

# 緊急のバグ修正が必要
git switch main
git checkout -b feature_B/hotfix

# バグ修正後、機能Aに戻る
git switch feature_A/func-a
    • 問題点：
      • ブランチ切り替えのたびにファイルの書き換えが発生
      • ビルドキャッシュや依存関係がリセットされる
      • 作業中の状態を保持できない
  • Worktree：
  • # image
.git/(リポジトリ)
├──       worktree A: /path/to/project/feature-a  (branch: feature_A)
├── (追加) worktree B: /path/to/project/hotfix     (branch: feature_B/hotfix)
└── (追加) worktree C: /path/to/project/main       (branch: main)

    # 現在の作業ディレクトリ（機能A開発中）
/path/to/project/feature-a $

# 緊急バグ修正用の新しいワークツリーを作成
git worktree add ../hotfix feature_B/hotfix

# 新しいワークツリーに移動
cd ../hotfix

# バグ修正作業...
git add .
git commit -m "Fix: critical bug"

# 機能Aの作業に戻る
# (編集中のファイル、ビルド環境などがそのまま保持されている)
cd ../feature-a

# ワークツリーの一覧表示
git worktree list

# ワークツリーの削除
git worktree remove <worktree-name>

# 全てのワークツリー状態確認
git worktree list --verbose

不要なら自然に GC される（孤立コミット）

ブランチを差し替え/作り直しする過程で、どのブランチ・タグからも参照されなくなったコミットは「孤立コミット (dangling / unreachable commits)」と呼ばれる。明示的に削除しなくても、Git が GC（garbage collection）で自動的に整理してくれる。

# どのブランチからも参照されなくなったコミット
[main] ----A----B----C--------------->
                  └─ X ─ Y ─ Z   ← 参照を失った orphan
                                  reflog のみが保持

GC のタイミング

Git は以下の条件で git gc (garbage collection) を実行する。

  1. 自動 GC: 多くの Git コマンド（commit, merge, fetch 等）の後に裏で gc --auto が走り、ゴミが溜まると自動実行
  2. reflog 期限切れ: デフォルトで 到達不能なコミットは 30 日後、到達可能なコミットは 90 日後に reflog から削除（gc.reflogExpireUnreachable / gc.reflogExpire）
  3. 手動実行: git gc --prune=now で即時削除

GC が走ると、reflog からも参照されなくなった孤立コミットは .git/objects/ から物理削除される。

つまり

  • 今すぐ消えるわけではなく、約 30 日かけて自動的に整理される
  • それまでは git reflog や直接 SHA 指定（git show <sha>）で復旧可能
  • ユーザーが何もしなくても、Git が勝手に掃除してくれるので 明示的な削除作業は不要
# 孤立コミットを確認
$ git fsck --lost-found

# reflog の期限設定を確認
$ git config --get gc.reflogExpireUnreachable  # default: 30 days
$ git config --get gc.reflogExpire             # default: 90 days

# 即時削除（強制）
$ git gc --prune=now

剪定中に置き忘れた札（index.lock）

git が書き込み操作（commit, add, checkout, rebase 等）を始めると .git/index.lock を作って排他制御する。正常終了時には自動で消えるが、エディタを Ctrl-C で殺した、ターミナルが落ちた、IDE/Vim プラグインが途中で止まった等で書き込みが中断されると lock ファイルが残骸として居残る。次に git を叩いた瞬間にこれが牙を剥く。

.git/
├── index            ← 本体（ステージング情報）
└── index.lock       ← 書き込み中フラグ。残ったままだと git が全部拒否

fatal: Unable to create '.../.git/index.lock': File exists.

Another git process seems to be running in this repository, e.g.
an editor opened by 'git commit'. Please make sure all processes
are terminated then try again. If it still fails, a git process
may have crashed in this repository earlier:
remove the file manually to continue.

削除前に「本当に stale か」を確認する

エラー文言通り rm するだけで直るが、現役の git プロセスが掴んでいる lock を消すとインデックス破損のリスクがある。削除前に以下 3 点を確認する。

# 1. lock ファイルの素性確認（タイムスタンプとサイズ）
$ ls -la .git/index.lock
-rw-r--r--  1 user  staff  0 May 20 09:21 .git/index.lock
#                          ↑ 0 byte           ↑ 30分前 → stale 濃厚

# 2. このリポジトリを掴んでいる git プロセスが居ないか
$ ps aux | grep -i git | grep -v grep
#   → エディタ・IDE 由来の長期プロセス、別タブの rebase 等が走っていないか確認
#   （statusline の `git status --porcelain` のような read-only は無関係）

# 3. 念のためエディタ・IDE 側で git 操作が止まっていないかも見る
#   （VSCode の Source Control パネル、Vim の fugitive、JetBrains の VCS 等）

削除して復帰 

$ rm .git/index.lock
$ git status   # 普通に動けば復旧完了

Warning

~/.gitconfig などで core.preloadIndex や独自ラッパーを設定している環境では、再発する場合がある。再発が頻発するなら、エディタの git 統合プラグイン（Vim の copilot/fugitive、VSCode 拡張など）が裏で git add を打っていないかを疑う。

兄弟 lock も覚えておく

index.lock 以外にも、同じ仕組みで書き込み中に作られる lock が複数ある。いずれも同じ要領で ls -la → プロセス確認 → rm で復帰可能。

Lock 何の操作で作られる 残った時の症状
.git/index.lock add, commit, checkout, rebase, merge 等の index 書き換え全般 ほぼ全ての git 操作がブロック
.git/HEAD.lock checkout, reset, commit 等の HEAD 更新 ブランチ切替・コミット不可
.git/refs/heads/<branch>.lock ブランチ ref 更新（branch -f, push 等） 該当ブランチの更新不可
.git/packed-refs.lock pack-refs（GC 中等） ref 操作系がブロック
.git/config.lock git config の書き換え 設定変更不可
.git/shallow.lock shallow clone の深度変更 fetch --deepen 等が失敗
