TCP/IP — 通信エラーに強くなる

データエンジニアは BigQuery、GCS、S3、Looker API、Composer のタスクなど、ほぼ全部「ネットワーク越し」に動かしている。

  • 「Connection reset by peer で DAG が落ちた」
  • 「bq load が途中で 503 を返した」
  • 「Looker のクエリだけ妙にタイムアウトする」

これらが「ネットワークのどの層で何が起きたのか」を読めるかどうかで、復旧スピードが何倍も変わる。

TCP/IP の超ざっくり地図

レイヤ 役割 データエンジニアが触る場面
アプリ層（HTTP/gRPC） 何を送るか bq CLI、gcloud、REST API、Looker API
トランスポート層（TCP） 確実に届ける コネクション確立、再送、輻輳制御
インターネット層（IP） どこへ届けるか ルーティング、サブネット、VPC
リンク層 物理 / L2 ほぼ触らない

Tip

エラーメッセージが「どの層」のものかを当てると原因の範囲が一気に絞れる。

TCP の挙動で知っておくべき4つ

(1) 3-way handshake（接続を張る） 

sequenceDiagram
    participant C as Client
    participant S as Server
    C->>S: SYN
    S->>C: SYN-ACK
    C->>S: ACK
    Note over C,S: Connection established
  • SYN → SYN-ACK → ACK の 3 回で接続を確立する
  • ここで失敗すると Connection refused（相手プロセスが居ない）/ timeout（そもそも到達しない）

(2) 再送（ロスっても粘る）

  • パケットが落ちると TCP は勝手に再送してくれる
  • だから「ネットワーク不調＝必ず即エラー」ではなく、遅いまま動き続けることがある

(3) コネクション切断

  • 正常終了は FIN、異常終了は RST
  • Connection reset by peer ＝相手側が RST を投げてきた、と読める

(4) タイムアウト

  • TCP のタイムアウトとアプリ側（HTTP クライアント、bq CLI 等）のタイムアウトは別物
  • アプリ側のタイムアウトのほうが短いことが多く、先に切るのはアプリ側

TCP/IP あるある（症状の読み方）

症状 TCP/IP 視点での読み
Connection refused 相手プロセス未起動 or ポート間違い。サービス疎通から確認
i/o timeout / connect timeout 到達はできていない疑い。Firewall / VPC-SC / NAT を確認
Connection reset by peer 相手が途中で切った。LB の idle timeout、サーバ側 OOM 等
503 Service Unavailable 到達はできた。アプリ層の問題（負荷、デプロイ中、レート制限）
bq load が途中で失敗 リトライで救えるケース多い。冪等性を担保した上で retry policy を設定

デバッグの最小ツールキット 

# 名前解決＋疎通確認
ping bigquery.googleapis.com           # ICMP（FW で塞がれていると無反応）
nc -vz bigquery.googleapis.com 443     # TCP の到達確認だけ
curl -v https://bigquery.googleapis.com  # アプリ層まで貫通

# どこで詰まっているかの切り分け
traceroute bigquery.googleapis.com     # 経路の可視化

Tip

切り分けは「内 → 外」「下層 → 上層」の順で。nc で TCP 通れば下層は OK、と一段ずつ確定する。

効きどころ

  • Composer (Airflow) の HttpOperator が時々落ちる → リトライポリシーと exponential backoff を設定する妥当性を判断できる
  • BigQuery への bq load で Connection reset → サーバ側の問題かネットワークかを切り分けて、retry で吸えるか判断できる
  • VPC-SC で外部 API が叩けない → IP 層の到達性問題と理解できれば、egress 設計の議論に乗れる

DNS — 名前解決で詰まらない

データエンジニアの仕事は、ほぼ全てが「ドメイン名で何かを叩く」。

  • bigquery.googleapis.com
  • looker.example.com
  • 社内 GitLab、社内 Airflow、社内 BI

このどれかで「名前解決できなくて止まる」事故が起きるとき、DNS の基本が分かっていれば一次切り分けが 5 分で終わる。逆に分からないと、「とりあえず再起動」「DNS キャッシュをクリア」を闇雲にやることになる。

DNS は何をしているか（1行）

Note

ドメイン名 → IP アドレスの変換表を、世界中の DNS サーバが分担して答える仕組み。

解決のステップ（クライアント目線） 

flowchart LR
    A[アプリ / OS] --> B{ローカルキャッシュ}
    B -->|hit| Z[IP 取得]
    B -->|miss| C{/etc/hosts}
    C -->|hit| Z
    C -->|miss| D[リゾルバ<br/>名前解決の代行サーバ<br/>社内DNS / 8.8.8.8 等]
    D --> E[ルート DNS]
    E --> F[TLD DNS]
    F --> G[権威 DNS]
    G -->|IP + TTL| D
    D --> Z
  1. ローカルキャッシュを見る（OS / ブラウザ / アプリ）
  2. /etc/hosts を見る（手動マッピング）
  3. リゾルバ（名前解決の代行サーバ：OS 設定の DNS サーバ、社内 DNS、8.8.8.8 等）に問い合わせる
  4. リゾルバは ルート → TLD → 権威 DNS とたどって IP を取り、TTL 付きでクライアントに返す

Tip

どのステップで止まったかを切り分けるのが、DNS 障害対応の本質。

知っておくべき5項目

(1) TTL（キャッシュの寿命）

  • DNS レコードには TTL がある（例: 300 秒）
  • レコードを切り替えても、TTL の間は古い IP を見続けるクライアントがいる
  • 「移行したのに半分だけ古い方を見ている」は大体これ

(2) A / AAAA / CNAME の違い

  • A = IPv4、AAAA = IPv6、CNAME = 別名（ドメイン → ドメイン）
  • BI ツールやプロキシ越しで CNAME 経由になっていると、解決チェーンが伸びる分だけ詰まりポイントが増える

(3) Private DNS / Cloud DNS

  • VPC 内では「外向き DNS」と別に「VPC 内専用の private zone」がある
  • BigQuery を private endpoint で叩く構成だと、bigquery.googleapis.com を private IP に解決させる private zone が前提

(4) /etc/hosts

  • 手元でだけ動く一時マッピング
  • 「自分の環境だけ動く / 動かない」の犯人になりがち。優先順位は hosts > DNS

(5) ネガティブキャッシュ

  • 「存在しない」も DNS は短期キャッシュする
  • 作りたて直後の名前を叩いて NXDOMAIN を引くと、しばらく失敗が残ることがある

DNS あるある（症状の読み方）

症状 DNS 視点での読み
自分の端末だけ社内 Looker に繋がらない リゾルバ設定 / VPN 経由の private DNS を確認
getaddrinfo: Name or service not known 名前解決そのものに失敗。typo、リゾルバ、ネットワーク到達のどれか
切り替え後も一部のジョブだけ古い接続先を見る TTL 内に張ったコネクションが生きている / クライアントキャッシュ
VPC-SC 配下で BigQuery が叩けない private endpoint への解決経路が無い疑い
自宅は OK・オフィスは NG ネットワークごとに違うリゾルバを使っていて、片方の zone に未登録

デバッグの最小ツールキット 

# 何をどう解決しているか
dig bigquery.googleapis.com           # A レコード
dig bigquery.googleapis.com +trace    # ルートからの解決経路
dig bigquery.googleapis.com @8.8.8.8  # 特定リゾルバに問い合わせ

# 軽量代替
nslookup bigquery.googleapis.com
host bigquery.googleapis.com

# 自分の端末が使っているリゾルバ
scutil --dns | head -30   # macOS

Tip

「自分のリゾルバで引けるか」「Google のパブリック DNS でも引けるか」を比較するだけで、問題が手元か / 社内 DNS か / 権威側かの3分割ができる。

効きどころ

  • VPC-SC や private endpoint の議論で、何が解決されて何が解決されないかを会話できる
  • マイグレーション時の TTL を意識し、リリース前に短くしておく提案ができる
  • 「自分の環境だけ動く」問題で、hosts / 社内 DNS / VPN の切り分けを即座にやれる

Caching — パフォーマンス理解が進む

データエンジニアの世界では「キャッシュ」が 5 つくらいのレイヤに重なって存在する。

  • 「データ直したのにダッシュボードが古いまま」
  • 「同じクエリなのに今日は爆速 / 明日は遅い」
  • 「BQ コストが急に増えた」

これらは大体「どのキャッシュが効いて / どのキャッシュが切れたか」を読み違えているのが原因。レイヤを地図化しておくと、即座に当たりがつく。

キャッシュの本質（1行）

Note

計算 or 取得を 1 回で済ませて、結果を「速い場所」に置いておく。

トレードオフは常に同じ：

  • ✅ 速い、安い
  • ❌ 古くなる（stale）、整合性が崩れる

データ基盤で出会うキャッシュレイヤ

レイヤ 主体 キャッシュの中身 失効の引き金
BigQuery クエリキャッシュ BQ 24h まで同一クエリ結果を返す クエリ文字列/参照テーブルの変更
BigQuery Materialized View BQ 集計済みデータ 自動 / 手動リフレッシュ
Dataform incremental Dataform 既存テーブルに差分追加 --full-refresh で再構築
Looker クエリキャッシュ Looker LookML クエリ結果 Datagroup の max_cache_age 経過
Looker PDT Looker 物理テーブルとして永続化 Datagroup or persist_for 期限
BI / ブラウザ ブラウザ ダッシュボード描画結果 リロード（強制リロード）
CDN CloudFront 等 静的アセット TTL 経過 / invalidation

Caching あるある（症状の読み方）

症状 怪しいキャッシュ
ETL は走ったのにダッシュボードが古い Looker クエリキャッシュ or PDT
同じクエリで料金 0 円が続いた BigQuery クエリキャッシュ（24h）
PDT がエラーで存在しない Datagroup 未トリガー（PDT がビルドされていない）
一部ユーザーだけ古い画面 ブラウザ / CDN キャッシュ
直したはずなのに古い結果 テストで使った中間テーブルがキャッシュされていた

Looker のキャッシュ三段論法

Looker は クエリキャッシュ + PDT + Datagroup が絡むので特に混乱しやすい。

flowchart LR
    U[ユーザー操作] --> Q{クエリキャッシュ<br/>max_cache_age}
    Q -->|hit| R[結果返却]
    Q -->|miss| P{PDT<br/>datagroup_trigger / persist_for}
    P -->|hit| R
    P -->|miss| B[BigQuery クエリ実行]
    B --> R

Warning

「キャッシュをクリアした」と言われたとき、どの層をクリアしたかを確認する習慣をつける。

キャッシュとの付き合い方（実務指針）

  1. 「古さの上限」を決めてから設計する

    • BI なら 1 時間遅れまで OK か、リアルタイムが要件か
    • 古さを許容できれば、コストもレイテンシも大幅に改善する

  2. キャッシュを「明示的にクリアできる経路」を必ず用意する

    • Looker: Admin → Clear Cache
    • BQ: クエリに無害な変更を入れて cache miss させる
    • Dataform: --full-refresh

  3. 障害対応では「キャッシュ層をスキップして真実を見る」

    • BQ なら -job_id で実際のクエリログを確認
    • Looker なら SQL Runner で直接叩く（PDT 経由しない）

  4. 「速くなった！」を喜ぶ前にキャッシュ起因を疑う

    • クエリキャッシュが効いているだけ、ということは多い
    • 計測は cold / warm 両方で

効きどころ

  • QA 環境で「直したのに反映されない」事故を即トリアージできる
  • PDT 未ビルドのワークアラウンド（persist_for 切替）を理解した上で提案できる
  • コスト急増の真因（キャッシュが切れて毎回 BQ にヒットしている等）を当てられる

Threads & Async — 処理の流れが見える

データエンジニアは「並列」を毎日触っているのに、実装の中身は意外と曖昧にしたまま動かしていることが多い。

  • Airflow の parallelism / max_active_runs をどう決めるか
  • BigQuery のジョブは「投げて待つ」非同期。.result() の前後で何が起きているか
  • dbt の threads を上げたら速くなるけど、上げすぎると詰まる
  • Python スクリプトで requests を並列化したい

これらを 「スレッド」「プロセス」「非同期」のどれで解いているかを区別できると、設計の解像度が一段上がる。

並列の 3 形態（超ざっくり）

形態 同時性 向いている処理
マルチプロセス 真の並列（CPU コア数まで） CPU バウンド（重い計算、前処理）
マルチスレッド OS が時分割（Python は GIL あり） I/O バウンド（API、DB、ファイル）
非同期 (async/await) シングルスレッド + 協調的切替 大量の I/O（HTTP, DB）

Note

Python のデータエンジニアが触るのは、ほぼ I/O バウンド。だから「スレッド」or「非同期」が中心。

同期 vs 非同期（1枚絵） 

gantt
    title 同期 vs 非同期（時間軸）
    dateFormat  X
    axisFormat  %s
    section 同期
    A 投げる    :a1, 0, 1
    A 待つ      :a2, after a1, 2
    B 投げる    :a3, after a2, 1
    B 待つ      :a4, after a3, 2
    C 投げる    :a5, after a4, 1
    C 待つ      :a6, after a5, 2
    section 非同期
    A 投げる    :b1, 0, 1
    B 投げる    :b2, after b1, 1
    C 投げる    :b3, after b2, 1
    A,B,C まとめて待つ :b4, after b3, 2

データエンジニアが触るほぼ全ての API（BigQuery、GCS、HTTP）は 「投げる」と「待つ」を分けられる非同期 I/F を持っている。

BigQuery を例に 

from google.cloud import bigquery

client = bigquery.Client()

# 1. ジョブを投げる（非同期 / すぐ返る）
job = client.query("SELECT COUNT(*) FROM `proj.ds.tbl`")
print(job.job_id, job.state)   # 'RUNNING'

# 2. ここで他の仕事を進められる

# 3. 結果が要るタイミングで待つ
rows = job.result()             # ここでブロック

Tip

「client.query() ＝ジョブ投入」「job.result() ＝待ち」。これを意識すると、複数クエリを同時投入 → まとめて待つ設計に切り替えられる。

Airflow / Composer の並列パラメータ

Airflow には「並列度の蛇口」が階層的にある。どれを上げ下げしているかを意識する。

パラメータ スコープ
parallelism クラスタ全体で同時実行できるタスク数
dag_concurrency 1 DAG 内で同時実行できるタスク数
max_active_runs 同じ DAG で同時に走れる run の数
pool 共有リソース（外部 DB 等）を奪い合う粒度の制御
task_concurrency 1 タスク種が同時に走れる数

Tip

「タスクが詰まっている」ときは、上から順に「どこで詰まっているか」を当てる。

dbt の threads

  • dbt は 依存グラフを並列に解いていく
  • threads は DWH に 同時投入する接続数
  • 上げると速いが、DWH 側のスロット / コネクション枠を奪うので、他のジョブを詰まらせることがある

Warning

「dbt が早くなった = 他のクエリが遅くなる」ことがある。BigQuery なら slot 競合、Snowflake なら warehouse 同時実行に注意。

並列で起きがちな事故

事故 原因 対処
同じテーブルに同時 INSERT 並列度を上げすぎ パーティション分割 / シリアライズ
API レート制限に引っかかる 同時リクエスト過多 バックオフ + 並列度の上限制御
デッドロック 複数リソースを違う順序で取得 取得順を統一
メモリ枯渇 並列ごとにバッファが積み上がる バッチサイズを下げる
ロギングが混線 複数スレッドが同じ stdout スレッド ID をログに含める

効きどころ

  • Composer の DAG 設計で、「直列で書くべきか / 並列で書くべきか」を根拠を持って判断できる
  • BQ ジョブを Python から大量投入するときに、job.result() の位置を最適化して全体時間を短縮できる
  • dbt の threads 調整で、速度と他ジョブへの影響を天秤にかけられる

Memory — 謎の挙動に強くなる

データを扱う以上、メモリは 「いつも触っているのに、見えないリソース」。

  • BigQuery が Resources exceeded during query execution を返す
  • pandas で MemoryError / Killed（OOM）が出る
  • Composer の worker が突然死して DAG が unstable になる
  • Looker の Explore で「メモリ上限を超えました」が出る

これらの「謎の挙動」は、メモリの使われ方の基礎モデルを持っていれば、原因の当たりをほぼ即座につけられる。

メモリの超ざっくりモデル

領域 何が入る 増え方
スタック 関数呼び出し / ローカル変数 浅い / 深いで決まる
ヒープ 動的に確保したデータ（DataFrame、配列等） 自分で del するか GC 任せ
ページキャッシュ (OS) ファイル I/O のキャッシュ OS が管理

Note

データエンジニアが詰まるのは ほぼ「ヒープに積み上がりすぎ」。

GC（ガベージコレクション）の前提

  • Python / Java / Go は GC 付き
  • GC は「もう参照されていないオブジェクトを掃除する」
  • だから 「参照が残っている限り解放されない」
  • pandas で「巨大な中間 DataFrame をローカル変数に残したまま次の処理」はメモリリーク的に見える

データ基盤あるある — 症状とメモリ視点

(a) BigQuery Resources exceeded

  • 原因: 単一ワーカーのメモリ上限超過。よくあるのは…
    • ORDER BY を巨大データに掛けた（全件をワーカーに集約）
    • WINDOW 関数の partition が広すぎる
    • ARRAY_AGG/STRING_AGG で巨大配列を作った
  • 対処: PARTITION BY を細かく / APPROX_TOP_COUNT 等の近似関数 / LIMIT をプッシュダウン

(b) pandas / Polars の OOM

  • 原因: ファイル全量を read_csv で 1 枚の DataFrame に
  • 対処:
    • chunksize でストリーム処理
    • dtype を明示（object → category、int64 → int32）
    • そもそも BigQuery で処理しきって、必要な分だけ pandas に持ってくる

(c) Composer / Airflow worker OOM

  • 原因: タスク内で巨大データを Python に展開（BigQueryHook.get_pandas_df の全件取得など）
  • 対処: 処理を BQ 側に寄せる / 結果セットの上限を設ける / worker のマシンタイプを上げる

(d) Looker Explore メモリ超過

  • 原因: 結果セットが Looker サーバのメモリに乗らない
  • 対処: フィルタを必須化する / row limit / PDT 化して事前集計

デバッグの最小ツールキット 

# プロセス単位のメモリ
ps aux | sort -rk6 | head -5     # RSS でソート上位
top                              # リアルタイム

# OS の状況
free -h                          # 全体メモリ（Linux）
vm_stat                          # macOS

# Python ローカル
python -m memory_profiler script.py   # 行ごとのメモリ
tracemalloc                          # ヒープのスナップショット

BigQuery 側は Query Plan と ジョブメタデータを見る。

bq show --format=prettyjson -j <JOB_ID>   # slot 消費・shuffle 量

設計指針 5箇条

  1. 重い処理は DWH（BigQuery）に寄せる
    • クライアント（pandas / Python）に全件持ってこない
  2. ストリーミング or バッチで「メモリに乗る粒度」を保つ
    • chunksize、for chunk in ...
  3. ORDER BY / WINDOW を巨大データにいきなり掛けない
    • 先に絞る / partition で分割
  4. dtype を意識する
    • 1 億行で int64 → int32 だけで半減
  5. メモリは可視化する
    • 「動いたから OK」ではなく「ピーク使用量はいくつか」を見る

効きどころ

  • Resources exceeded の根本対処で、ORDER BY の前に絞れる SQL に書き換えられる
  • pandas OOM で、「マシンを大きくする」前に「dtype と chunk」を提案できる
  • Composer の不安定さを見たとき、worker のメモリプレッシャーを疑える
