[Looker] 全体概要

[!NOTE] データの流れ:

データは BQ → view → explore→ dashboard の順に流れる。
ディレクトリ構成もこの流れに沿って上流から下流へ並べて示す。

  1. BQ — BigQuery上の実テーブル（データソース）
  2. View — BQテーブルを参照し、dimension/measureを定義
  3. Explore — Viewを結合して、ユーザーが探索できる単位を作る
  4. Dashboard — Exploreから取得したデータをタイル（グラフ等）で描画

厳密に言えば、間に Model が挟まる（ModelがBQ接続を指定し、Exploreをユーザーに公開する役割）。
完全に書くと BQ → View → Model(Explore定義) → Dashboard。
ただし、Modelは「配線役」であってデータ変換の層ではないので、データの流れという観点では BQ → View → Explore → Dashboard で十分。

 my_looker_project/
  │
  │  ── プロジェクト設定【必須】──────────────────────────────
  │
  ├── manifest.lkml          ← BQ接続先のデータセット名・定数を定義
  ├── manifest_lock.lkml      （自動生成、手動編集不要）
  │
  │  ── [1] View 定義層【必須】（上流：BQ テーブル参照）──────
  │         sql_table_name で BQ テーブルを直接参照、
  │         または derived_table で複数テーブルを結合・加工。
  │         dimension（属性）・measure（集計値）を定義する。
  │
  ├── views/
  │   ├── orders.view.lkml
  │   ├── users.view.lkml
  │   └── products.view.lkml
  │
  │  ── [2] Explore 定義層【任意】（中流：view を結合）───────
  │         view を JOIN し、ユーザーがデータ探索できる単位を作る。
  │         ※ view ファイル内で explore を定義するケースもある
  │
  ├── explores/
  │   └── order_analysis.explore.lkml
  │
  │  ── [3] Model 定義層【必須】（BQ と Explore の紐付け）────
  │         connection で BQ プロジェクトを指定し、
  │         どの Explore をユーザーに提供するかを宣言する。
  │
  ├── models/
  │   └── *.model.lkml
  │
  │  ── [4] Dashboard 描画層【任意】（下流：BI 表示）─────────
  │         explore を参照し、タイルごとに
  │         dimension / measure / filter を選んで描画する。
  │
  ├── dashboards/
  │   └── sales_overview.dashboard.lookml
  │
  │  ── その他【任意】──────────────────────────────────────
  │         データテストやドキュメントなど、プロジェクトに必要なファイルを配置。
  │
  ├── data_tests/
  │   └── tests.lkml
  └── documents/
      └── README.md
Lookerのプロジェクト構成例（最小版）

[!note]

 my_looker_project/
  ├── manifest.lkml          # プロジェクト設定（モデル名、ロケールなど）
  ├── sample.model.lkml      # モデルファイル（DB接続、Explore定義）
  └── views/                 # View ファイル（テーブル定義）
      └── orders.view.lkml

Explore画面からデータを探索できるように、 Model で View を include し、Explore を定義すればOK！

[Looker] 個別ファイル

[!WARNING] Syntaxの関係上、拡張子を.yamlとしているが、実際には.lkmlである点に注意

connection: "your_connection_name"
include: "/views/*.view.lkml"
include: "/models/*.model.lkml"
include: "/dashboards/*.dashboard.lkml"

view: sample_view {
  dimension: id {
    type: number
    sql: ${TABLE}.id ;;
  }

  measure: count {
    type: count
    sql: ${TABLE}.id ;;
  }
}

[!TIP]
Looker = セマンティックレイヤー と呼ばれる理由

Lookerはよくセマンティくレイヤーと言われるが、view定義をモリモリにするとそうなる
View にビジネスロジックを集約する設計思想に由来。

  • measure で「売上」「解約率」など指標の計算式を一元定義
  • dimension に label / description で業務用語を付与
  • derived_table でテーブル結合・加工ロジックを吸収
  • drill_fields でドリルダウン先を定義

「“売上” とはこの SQL である」という定義を View 層に持たせることで、 ダッシュボードごとに SQL を書き直す必要がなくなり、全社で指標の定義が統一される。

explore: order_analysis {
  from: sample_view
  join: another_view {
    type: left_outer
    sql_on: ${sample_view.id} = ${another_view.sample_view_id} ;;
  }
}

model: sample_model {
  explore: sample_view {
    join: another_view {
      type: left_outer
      sql_on: ${sample_view.id} = ${another_view.sample_view_id} ;;
    }
  }
}

dashboard: sample_dashboard {
  title: "Sample Dashboard"
  layout: {
    type: "grid"
    columns: 12
  }
element: {
    title: "Sample Tile"
    type: "lookml_tile"
    lookml_tile: {
        model: "sample_model"
        explore: "sample_view"
        fields: ["sample_view.id", "sample_view.count"]
        filters: {}
        visualization: {
            type: "bar"
        }
    }
}

[Looker] 用語集

WebUI

  • Explore (エクスプローラ)
    • -> データの探索や分析を行うためのインターフェース
  • Tile (タイル)
    • -> ダッシュボード上で表示される個々のグラフやチャートのコンポーネント
  • LookML Dashboard (ルックエムエル ダッシュボード)
    • -> LookMLで定義されたダッシュボード
  • Looker Blocks (ルッカーブロック)
  • Looker Marketplace (ルッカーマーケットプレイス)

LookML（基礎）

  • Model (モデル)
    • -> 複数のViewを組み合わせてExploreを定義する
  • View (ビュー)
    • -> データベースのテーブルや派生テーブルを表現する
  • Dimension (ディメンション)
    • -> データの属性やカテゴリを表現する
  • Measure (メジャー)
    • -> 集計や計算を表現する
  • Filter (フィルター)
    • -> データの絞り込み条件を定義する
  • Parameter (パラメーター)
    • -> ユーザー入力を受け取るための変数を定義する
  • Set (セット)
    • -> 複数のディメンションやメジャーをグループ化する
  • Explore (エクスプローラ)
    • -> データの探索や分析を行うためのインターフェース
  • Join (ジョイン)
    • -> 複数のViewを結合してデータを取得する
  • SQL (エスキューエル)
    • -> LookML内で直接SQLクエリを記述することができる
  • Liquid (リキッド)
    • -> LookML内で動的なコンテンツを生成するためのテンプレート言語
  • YAML (ヤムル)
    • -> LookMLの構造や設定を定義するためのデータシリアライゼーション形式

LookML（応用）

  • DT (Derived Table: 派生テーブル）

    • cf. Lookerの派生テーブル Google Cloud Looker

  • NDT (Native Derived Table: ネイティブ派生テーブル)

    • -> ViewやExploreを一意に再利用でき、LookML記述でSQLを書き出せるORM的なイメージ
    • cf. ネイティブ派生テーブル（NDT）の活用方法 Google Cloud Looker Community
    • cf. ネイティブ派生テーブルの作成 Google Cloud Looker

  • PDT (Persistent Derived Table: 永続的な派生テーブル)

    • cf. 永続的な派生テーブル Google Cloud Looker

[Looker] 分析

⏺ データキューブとは

データキューブは、多次元でデータを分析するための概念モデルです。

基本的な考え方

通常のテーブル（2次元）を、3次元以上に拡張したものをイメージしてください。

      [時間軸]
         |
         |
         +------[地域軸]
        /
       /
  [商品軸]

各軸（= ディメンション）の交点にあるセルに、売上額や件数などのメジャー（集計値） が入ります。

具体例

例えば売上データなら：

ディメンション 例
時間 2025年1月, 2月, …
地域 東京, 大阪, …
商品カテゴリ 食品, 家電, …

この3軸の任意の組み合わせで「売上合計」や「件数」をスライスして見られる、というのがデータキューブの考え方。

Looker との関係

Lookerでは Explore がデータキューブに相当します。

  • View でディメンションとメジャーを定義
  • Explore でそれらを結合し、ユーザーが自由に軸を選んで集計できる状態にする

つまり「データキューブを設計する」とは、Lookerの文脈では どのディメンションとメジャーを用意し、どうjoinして Explore として公開するか を決めることです。

OLAP操作との関連用語

データキューブに対する操作として、以下の用語もよく使われます：

  • スライス — 1つの軸を固定して断面を見る（例：東京だけ）
  • ダイス — 複数軸で範囲を絞る（例：東京×食品）
  • ドリルダウン — 粒度を細かくする（例：年→月→日）
  • ロールアップ — 粒度を粗くする（例：日→月→年）

チームで「データキューブ」と言われたら、「この分析テーマについて、どんな切り口（ディ メンション）と集計値（メジャー）を用意するか」という設計の話だと理解すれば大丈夫。
