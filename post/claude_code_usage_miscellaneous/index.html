Subagent（Task tool）

単発の作業を委任する仕組み。（普段の会話で自然に使われている）

あなた: 「このプロジェクトのルーティング処理がどこにあるか調べて」

Claude（内部処理）: → 「探索タスクだな、Exploreエージェントに任せよう」 → Task tool で Explore サブエージェントを生成 → 結果を受け取って、あなたに回答

普通に会話しているように見えますが、裏でSubagentが動いてくれていいる。

明示的に指示するパターン

  特定のやり方を指定したい場合は、自然言語で伝えるだけ。

  # 並列で調査させたいとき
  「フロントエンドとバックエンドのエラーハンドリングを並列で調べて」

  # チームで作業させたいとき
  「チームを組んで、このリファクタリングを進めて」

  # 特定のエージェントタイプを使わせたいとき（上級者向け）
  「Bash エージェントでテストを回して」

Subagentの種類（参考）

  ┌─────────────────┬──────────────────────────────┐
  │      種類       │          得意なこと          │
  ├─────────────────┼──────────────────────────────┤
  │ Explore         │ コード検索・構造調査         │
  ├─────────────────┼──────────────────────────────┤
  │ Plan            │ 実装方針の設計               │
  ├─────────────────┼──────────────────────────────┤
  │ Bash            │ コマンド実行                 │
  ├─────────────────┼──────────────────────────────┤
  │ general-purpose │ 何でも（ファイル編集も可能） │
  └─────────────────┴──────────────────────────────┘

Claude: 「この調査はサブエージェントに任せよう」 → Task tool で1体のエージェントを生成 → 結果を受け取って終了（使い捨て）

  • 1つのタスクを投げて、結果を返してもらう
  • エージェント同士の通信はない
  • 呼び出し元が結果を統合する
  • 普段の会話でも頻繁に使われている（Explore, Plan 等）

Agent Teams

Subagentを複数束ねて、協調させる仕組み。

あなた：1つのセッションで「チームで作業して」と指示するだけ。

Claude (リーダー): → TeamCreate でチーム作成 → Task tool で複数の teammate を生成（=各々がSubagent） → 共有タスクリスト + SendMessage で相互連携 → リーダーが管理・統合

  • 複数のSubagentがチームとして動く
  • 共有タスクリスト（TaskCreate/TaskList）で作業を管理
  • SendMessageでエージェント間通信ができる
  • ライフサイクルが長い（複数タスクをこなす）

両者は実は同じ仕組みの上に成り立っているもので、使い分けのレベルが異る。

関係図:

  Agent Teams
   └── teammate A  ← Subagent（Task tool で生成）
   └── teammate B  ← Subagent（Task tool で生成）
   └── teammate C  ← Subagent（Task tool で生成）
        ↕ SendMessage で相互通信
        ↕ TaskList を共有

一言でまとめると

  ┌────────────┬────────────────────────┬────────────────────────┐
  │            │        Subagent        │      Agent Teams       │
  ├────────────┼────────────────────────┼────────────────────────┤
  │ 規模       │ 1体                    │ 複数体                 │
  ├────────────┼────────────────────────┼────────────────────────┤
  │ 通信       │ なし（結果を返すだけ） │ あり（メッセージ交換） │
  ├────────────┼────────────────────────┼────────────────────────┤
  │ タスク管理 │ なし                   │ 共有タスクリスト       │
  ├────────────┼────────────────────────┼────────────────────────┤
  │ 用途       │ 単発の調査・作業       │ 複雑な協調作業         │
  └────────────┴────────────────────────┴────────────────────────┘

Subagentは部品、Agent Teamsはそれを組み合わせたチーム体制という関係

Claude Code MCP設定ガイド

設定ファイルの優先順位（強い順）

優先度 スコープ ファイル
1 Managed /Library/Application Support/ClaudeCode/managed-mcp.json
2 Local ~/.claude.json > projects.{パス}.mcpServers
3 Project プロジェクトルートの .mcp.json
4 User ~/.claude.json > グローバル mcpServers
  • 同名サーバーは上位スコープが完全に勝つ（マージされない）
  • 名前が異なればすべて共存する

各ファイルの役割

~/.claude/settings.json

  • 環境変数、モデル設定、プラグイン設定など
  • MCPの定義は含まない

~/.claude.json

  • Claude Codeの統計情報・プロジェクト管理情報
  • MCP設定は2箇所に書ける（スコープが異なる）
    • mcpServers（ルート直下）→ User スコープ：全プロジェクト共通
    • projects.{パス}.mcpServers → Local スコープ：特定プロジェクトのみ
  • 注意: Claude Codeプロセスが随時書き込むため、手動編集時は競合に注意

.mcp.json（プロジェクトルート）

  • Project スコープ：そのディレクトリ配下で起動した場合に有効
  • VCS管理でチーム共有が可能
  • ${VAR} 形式の環境変数展開をサポート

現在の構成

MCP定義は ~/Sandbox/MDE/.mcp.json に集約済み。 ~/Sandbox/MDE/ 配下で起動した場合のみ有効。

// ~/Sandbox/MDE/.mcp.json
{
  "mcpServers": {
    "slack":      { ... },  // HTTP接続
    "mcp-luffy":  { ... },  // stdio (npx)
    "atlassian":  { ... },  // stdio (uvx)
    "github":     { ... },  // stdio (npx)
    "gitlab":     { ... }   // stdio (npx)
  }
}

.claude/settings.jsonと.claude/settings.local.json` の違い

settings.json settings.local.json
用途 チーム共有のプロジェクト設定 個人用のローカル設定
Git コミット対象 自動で gitignore される
典型的な内容 チーム共通のMCPサーバー、権限ポリシー、hooks 個人のMCP追加/除外、マシン固有設定

優先順位（高い順）：

  1. Managed settings（組織管理）
  2. コマンドライン引数
  3. .claude/settings.local.json（ローカル）
  4. .claude/settings.json（プロジェクト）
  5. ~/.claude/settings.json（ユーザーグローバル）

ローカル設定がプロジェクト設定より優先されるので、 たとえばチームで共有しているMCPサーバーを個人的に無効にしたい場合は settings.local.json で上書きできる。

MCPの使い分けに活用する場合：

  • チーム全員が使うMCPサーバー → settings.json に定義
  • 自分だけ追加/無効にしたいMCPサーバー → settings.local.json で上書き

この仕組みを使えば、普段不要なMCPサーバーをローカル設定で無効にしておき、 必要なときだけ有効に戻す運用ができる。

セットアップ プロンプト

ccstatuslineをインストールして、見やすい監視機能を設定してください。

以下の手順で実行してください：

1. ~/.claude/settings.json を確認して、現在の設定を表示
2. ccstatuslineの設定を追加（bunx ccstatusline@latestを使用）
3. 以下のウィジェットを含む推奨設定を提案：
   - モデル名
   - コンテキスト使用率（使用可能率）
   - セッションコスト
   - ブロックタイマー
   - 入力/出力トークン数
   - Gitブランチ

4. 設定を保存してClaude Codeの再起動方法を教えてください

元に戻す手順

# 復元
cp ~/.claude/settings.json.bak ~/.claude/settings.json
cp ~/.config/ccstatusline/settings.json.bak ~/.config/ccstatusline/settings.json
