[Docker] デーモン (daemon) とアーキテクチャ

Docker はクライアント・サーバー型のアーキテクチャを採用している。

docker CLI  →  docker.sock  →  dockerd (デーモン)  →  コンテナ操作
  • dockerd（Docker デーモン）: Docker Engine の本体。バックグラウンドで常駐し、コンテナの作成・実行・管理を行う
  • /var/run/docker.sock: CLI とデーモン間の通信路（Unix ソケット）。1つのデーモンだけがこのソケットを listen する
  • /var/run/docker.pid: プロセス ID を記録し、デーモンの二重起動を防ぐ（ロックファイル）
  • docker run 等の CLI コマンドは、このデーモンに API リクエストを送っているだけ
  • Docker Desktop を使っている場合は、アプリが裏で dockerd を起動・管理している

daemon の由来 コンピュータ用語としての daemon は 1963年の MIT Project MAC が起源。 ギリシャ神話の daimon（ダイモーン）＝善悪どちらでもない超自然的な仲介者・守護霊に由来する。 キリスト教で悪魔 (demon) に転じたのは後の話で、IT での意味は「裏で目に見えず働く存在」。 熱力学の思考実験「マクスウェルの悪魔」（分子を仕分けする見えない存在）からの着想とも言われている。

[Docker] コマンド一覧

  • Docker File

    • $ docker build . Dockerfileの格納されているディレクトリ上で実行（. はcdの意）
    • $ docker build -t {name} {directory} 名前指定してビルド

  • Docker Image

    • $ docker images イメージ一覧
    • $ docker pull {image} イメージ取得（eg. DockerHub → DockerImage）
    • $ docker build {directory} イメージ化（DockerFile → DockerImage）
    • $ docker run -it {image} bash イメージ実行（コンテナ起動）　※「pull」「start」を内部的に実行
      • $ docker run -it --rm {image} bash コンテナ起動後に削除
      • $ docker run -it -v {host}:{container} {imege} ホストのファイルシステムをコンテナにマウント
        • e.g. 
          $ docker run -it -v ~/host/mounted_folder:/new_dir {image} bash
      • docker run -u {UserId}:{UserGroup} ユーザーID、グループ名を指定してコンテナ作成 
        $ docker run -it -u $(id -u):$(id -g) -v ~/host/mounted_folder:/new_dir {image} bash
        • $ id -u PCのユーザーID確認
        • $ id -g PCのグループ名確認
      • $ docker run -p {host_port}:{container_port} ホストのポートをコンテナポートに繋げる 
        $ docker run -it -p 8888:8888 --rm jupyter/datascience-notebook bash
    • $ docker --cpus {# of CPUs} コンテナがアクセスできるCPUI上限確認
    • $ docker --memory {byte} 
      $ docker run -it --rm --cpus 2 --memory 2g ubuntu bash
      • $ sysctl -n hw.physicalcpu_max 物理コア数
      • $ sysctl -n hw.logicalcpu_max 論理コア数
      • $ sysctl hw.memsize メモリ（byte）
    • $ docker rmi {image} イメージ削除
    • $ docker rmi $(docker images -q) 全イメージ削除

  • Docker Container

    • $ docker ps -a コンテナ一覧（ps=process status）
    • $ docker inspect {container} コンテナのあらゆる情報確認 
      $ docker inspect 5f90be76cd31 | grep -i cpu
      • CPU数やメモリ量等確認時 （|grep=抽出 -i=ignore大文字小文字問わず {検索語句} ）
    • $ docker restart {container} コンテナ再起動
    • $ docker exec -it {container} bash コンテナ実行
    • $ docker stop {container} コンテナ停止
    • $ docker stop $(docker ps -q) 全コンテナ停止
    • $ docker rm {container} コンテナ削除
    • $ docker system prune、docker rm $(docker ps -q -a) 全コンテナ削除
    • $ docker run --name {name}{imagename} コンテナ名付け
    • $ docker run -d {imagename} detached mode コンテナ起動後にdetachする（バックグラウンドで動かす）
    • $ docker run --rm {imagename} foreground mode コンテナをExit後に削除する（使い捨てコンテナ用）
    • $ docker commit {imageid/name} {new_imagename(:tag)} コンテナ更新
    • $ docker tag {new_imagename(:tag)} {target} コンテナ名変更
    • $ docker push {imagename} コンテナをDockerHubへ

[Docker] Dockerfile

概要

ローカルやクラウドで、データ分析環境を容易に構築するためのDockerfile

# Host (Local or AWS/GCP)
docker/
  ┝ dsenv_build/
  │ └ Dockerfile  : ここ作る
  └ my-lab/       : 任意名

# Docker Containar
(root)/
  ┝ bin/
  ~
  ┝ opt/
  │ └ anaconda3
  └ work/          : 「my-lab」と紐づいている
                    (eg. Jupyterlabなどで分析「test.ipnb」などが格納される)
作成 
FROM ubuntu:latest
RUN apt-get update && apt-get install -y \
    sudo \
    wget \
    vim

WORKDIR /opt
RUN wget https://repo.anaconda.com/archive/Anaconda3-2019.10-Linux-x86_64.sh && \
    sh Anaconda3-2019.10-Linux-x86_64.sh -b -p /opt/anaconda3 && \
    rm -f Anaconda3-2019.10-Linux-x86_64.sh
ENV PATH /opt/anaconda3/bin:$PATH

WORKDIR /
RUN pip install --upgrade pip

CMD ["jupyter", "lab", "--ip=0.0.0.0", "--allow-root", "--LabApp.token=''"]
  • nb.
    • $ jupyter lab で起動
    • --ip 0.0.0.0 ローカルホストの意味
    • --allow-root ローカルで実行の為ルート権限（※ 別サーバーでならユーザー指定するべき）
    • --LabApp.token トークン指定でセキュリティを高める。（ローカル上の為未指定）
    • 書き方勘所

      1. FROM
      FROM {dockerimage} # OS等々指定
      1. RUN
      RUN {linux comand} # やりたいこと
      • eg.
      RUN touch test
RUN echo `hello world` > test
      • RUNの他に、COPY、ADDがレイヤーを作成するインストラクション
      • &&: コマンド結合
      • \: 改行
      • apt: ubuntuのパッケージ管理(パッケージインストール)
        • RUN apt-get install {package}
        • RUN apt-get update: 最新版取得
      • (注意) RUN を複数書きすぎてファイルが重くなる問題　
      • → Dockerimageのレイヤー数は最小に！
      • eg. レイヤー数削減 
        # Layer数:4 = RUN が4行
FROM ubuntu:latest
RUN apt-get update
RUN apt-get install XXX
RUN apt-get install YYY
RUN apt-get install ZZZ

        # Layer数:2
FROM ubuntu:latest
RUN apt-get update
RUN apt-get install XXX YYY ZZZ

        # Layer:1
# コマンドをまとめ、改行で見やすく
FROM ubuntu:latest
RUN  apt-get update && apt-get install \
     XXX \
     YYY \
     ZZZ
      1. CMD コンテナのデフォルトコマンドを指定。
      # Dockerfileの最後に記述（原則）
CMD ["executable", "param1", "param2"]

実行 

$ docker run \
    -p 8888:8888 \
    -v ~/docker/ds_python/:/work \
    --name my-lab 511a925b20aa
  • -p publishしてコンテナとホストのポートを設定
  • -v ホストとコンテナ内部のディレクトリを共有
    • ~/docker/ds_python/:/work {ホスト内}:{コンテナ内}
  • --name: dockerimage に名前を作成
分析環境構築 
# ディレクトリ作成 & 移動
$ mkdir docker && cd docker

# Dockerイメージ作成してコンテナ起動
$ docker build .
$ docker run \
    -p 8888:8888 \
    -v ~/docker/ds_python:work \
    --name my-lab {image}

# SSHでクラウドに接続（セキュアにログイン）
$ ssh -i hoge.pem ubuntu@ec2-00-100-200-300.ap-northeast-1.compute.amazonaws.com
  • nb.
  • 予めAWSに（EC2サーバー）インスタンスを作成して、そこにログインしておく。
  • hoge.pem AWSのアクセス認証鍵で名前は任意。（鍵のあるディレクトリで実行）
  • ec2-{~}.compute.amazonaws.com PublicDNSは都度取得。
# クラウド上にDockerCE（=ComunicatEdition）をインストール
(aws)$ sudo apt-get update
(aws)$ sudo apt-get install docker.io
(aws)$ docker --version
(aws)$ sudo gpasswd -a ubuntu docker  # sudo無しでdockerコマンド使用する為


# Dockerimageをクラウドに送る
# [case.1] クラウドがネットワークに繋がれる
$ sftp -i hoge.pem ubuntu@ec2-00-100-200-300.ap-northeast-1.compute.amazonaws.com
(aws)$ put ~/docker/dsenv_build/Dockerfile home/ubuntu

# buildcontextを作成して、ファイル移動
$ ssh -i hoge.pem ubuntu@ec2-00-100-200-300.ap-northeast-1.compute.amazonaws.com
(aws)$ mkdir dsenv_build
(aws)$ mv Dockerfile dsenv_build
(aws)$ cd dsenv_build
(aws)~/dsenv_build$ docker build .

# dockerコンテナ、JupyterLab起動。
(aws)~dsenv_build$ docker run -v ~:/work -p 8888:8888 {imageID}
# ブラウザーからアクセスする。
# `@ec2-00-100-200-300.ap-northeast-1.compute.amazonaws.com:8888`

# [case.2] クラウドがネットワークに繋がれない
# ローカルでDockerimageをtarで圧縮
$ docker build .
# $ docker build --platform linux/amd64. # nb. MacのM1チップ使用時は
$ docker save {imageID} > myimage.tar

$ sftp -i hoge.pem ubuntu@ec2-00-100-200-300.ap-northeast-1.compute.amazonaws.com
(aws)$ put ~/docker/myimage.tar home/ubuntu
(aws)$ docker load < myimage.tar # tarファイルを解凍
(aws)$ docker run -it {imageID} bash

[Docker Compose]

[Docker Compose] Ollama + Open WebUI 環境構築手順（実践）

  • ollama : Local LLMs Made Easy
  • Open WebUI : The open source web interface for LLMs cf. https://qiita.com/kkawaharanet/items/b80d7cde49364aa94963

概要 

# 作業ディレクトリ
ollama-setup/
  ├── docker-compose.yml  : Docker Compose 設定ファイル
  ├── open_webui_data/    : Open WebUI のデータ保存用ボリューム
  └── ollama_data/        : Ollama のデータ保存用ボリューム

# ネットワークの仕組み

 💻 あなたのMac (localhost)
  ├── ポート 3000  →  open-webui コンテナ :8080
  ├── ポート 11434 →  ollama     コンテナ :11434
  │
  └── Docker内部ネットワーク (ollama-setup_default)
      ├── open-webui コンテナ
      │   └── http://ollama:11434 で Ollama にアクセス
      └── ollama コンテナ

services:                                      # 複数のコンテナを定義するセクション

  # [サービス1]: Ollama (LLM実行エンジン)
  ollama:
    image: ollama/ollama:latest                # Docker Hubから取得するイメージ
    container_name: ollama                     # コンテナ名を"ollama"に固定
    ports:
      - "11434:11434"                          # ポートマッピング（ホスト:コンテナ）
    volumes:
      - ollama_data:/root/.ollama
    environment:
      - OLLAMA_INSECURE=true                   # データ永続化
    restart: unless-stopped                    # 自動再起動設定

  # [サービス2]: Open WebUI (ブラウザUI)
  open-webui:
    image: ghcr.io/open-webui/open-webui:main  # GitHub Container Registryから取得
    container_name: open-webui
    ports:
      - "3000:8080"                            # Mac側:3000 → コンテナ内:8080
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434    # Ollamaへの接続先
    volumes:
      - open_webui_data:/app/backend/data      # チャット履歴などを保存
    depends_on:
      - ollama                                 # ollamaコンテナを先に起動
    restart: unless-stopped

volumes:                                       # データ永続化用のボリューム定義
  ollama_data:
  open_webui_data:

実装 (Docker Compose 使用) 

# [1] 環境構築
cat << EOF > docker-compose.yml
services:
  ollama:
    image: ollama/ollama:latest
    container_name: ollama
    ports:
      - "11434:11434"
    volumes:
      - ollama_data:/root/.ollama
    environment:
      - OLLAMA_INSECURE=true
    restart: unless-stopped

  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    container_name: open-webui
    ports:
      - "3000:8080"
    environment:
      - OLLAMA_BASE_URL=http://ollama:11434
    volumes:
      - open_webui_data:/app/backend/data
    depends_on:
      - ollama
    restart: unless-stopped
EOF

# [2] Dockerコンテナの起動
docker compose up -d

# [3] Ollamaモデルのダウンロード
# コンテナ内でコマンド実行
# docker exec -it ollama bash   # ollamaコンテナ内でbash起動

docker exec -it ollama ollama pull llama3.2

# [4] Open WebUIにアクセス
open http://localhost:3000

環境（OllamaとOpen WebUI）の起動・停止

│        目的         │        コマンド        │
├─────────────────────┼────────────────────────┤

├─────────────────────┼────────────────────────┤
│ Mac再起動前に片付け │ `docker compose down` -> `docker compose up -d`
├─────────────────────┼────────────────────────┤
│ 完全にやり直したい  │ `docker compose down -v` -> `docker compose up -d`
└─────────────────────┴────────────────────────┘

# [監視] リソース確認
docker images                 # ダウンロード済みイメージ一覧
docker volume ls              # ボリューム一覧
docker system df              # ディスク使用量
docker logs ollama            # ollamaコンテナのログ
docker logs -f open-webui     # リアルタイムでログ表示（-f: follow）

代替（Docker経由ではなく、Mac上に直接Ollamaをインストールする方法も） 


# Homebrewでインストール
brew install ollama

# Ollamaサービス起動
ollama serve &

# モデルダウンロード
ollama pull llama3.2

代替構成例（OllamaをMac上で直接実行し、Open WebUIをDockerで起動） 

# Homebrewでインストール
brew install ollama

# バックグラウンドで起動
brew services start ollama

# モデルダウンロード
ollama pull llama3.2

# Open WebUIだけDockerで起動（既存の設定で動く）
cd ~/ollama-setup
docker compose up -d open-webui

この構成:

  • Ollama: Mac上で直接実行（証明書問題なし）
  • Open WebUI: Docker コンテナ（http://localhost:3000でアクセス可能）
